Unit Testing

Unit testing is a level of software testing in which individual units are tested in order to validate that each section works as expected. A unit is the smallest part of a system, normally has only a few inputs and a single output.

Unit testing is the first form of testing that normally takes place on a system and is performed prior to intergration testing. It uses the white box testing method as the person who writes the tests are normally aware of the interal working of the system.

Unit testing frameworks, drivers, stubs, and mock/ fake objects are used to assist in unit testing.

In procedural programming a unit may be an individual program, function, procedure, etc.In object oriented programming the smallest unit of a program is a method which will belong to some form of class.

Advantages

  • Increase confidence to change, refactor and maintain code
  • Increases development speed by allowing methods to be tested early in the process, which increases the number of bugs caught early.
  • Debugging is easy as when tests fail it is highly likely a recent change is the issue

Disadvantages

  • Does not prove absence of errors
  • If initial tests are not correct their accuracy cannot be guaranteed

White Box Testing

White box testing (clear box testing, glass box testing, transparent box testing, structural testing) looks at the software been tested and uses the knowledge as part to the test. White-box testing requires internal knowledge of the system and programming skills. It is useful as it provides an internal perspective of the software under test.

Advantages

  • Efficient in finding errors and problems
  • Knowledge of internals is beneficial for thorough testing
  • Allows finding hidden errors
  • Programmers introspection
  • Helps optimizing the code
  • Due to required internal knowledge of the software, maximum coverage is obtained

Disadvantages

  • Might not find unimplemented or missing features
  • Requires high level knowledge of internals of the software under test
  • Requires code access

Black Box Testing

Black box testing (functional testing) is a method by which software is tested without its internals being known. The tests use the software interfaces and try to ensure they work as expected. The tester will be aware of how the software should function but know the processes which result in the functionality.

Advantages

  • Efficient for large segments of code
  • Code access is not required
  • Separation between user’s and developer’s perspectives

Disadvantages

  • Limited coverage since only a fraction of test scenarios is performed
  • Inefficient testing due to tester’s luck of knowledge about software internals
  • Blind coverage since tester has limited knowledge about the application

Dynamic testing

Dynamic testing is the opposite of static testing and involves testing code by running it, providing input and comparing the result against the expected value. Types of dynamic testing include unit testing, integration testing, system testing and acceptance testing.

Static Testing

Static testing (the opposite of dynamic testing) is a software testing method which involves testing the software without running the actual program. Static testing may conducted manually or with the aid of software and may include code analysis, inspection, code reviews and walkthroughs.

Errors found using this method may include undefined variables, unused code, unreachable code, security vulnerabilities, standards violations or syntax violations.
