Removing Commits

To remove the previous commit use the command:

git reset --hard HEAD^

To remove one or more commits use the command:

git reset --hard HEAD~n
 (n is the number of commits to remove)

Branches

A branch in git is a way of creating multiple paths for a project. These allow you to work on different sections of a project separetly and, once the desired task is complete, can be merged into the main project.

git branch
This command will print to the terminal all the branches of the current repository. If no others have been made only the main branch known as master will be shown.

git branch branch-1
git checkout -b branch-1
To create a new branch call git branch followed by the desired branch name (which should not include spaces). This will create a branch that matches to current one. Alternativly you may use the command checkout with the option -b, which will create a new branch and switch to it.

git checkout branch-1
git checkout branch-1 -f
Use this command to move between branches that are being worked on. If there are uncommited changes when this is called git should present a warning message and not switch branches. This can be ignored using the option -f.

git branch -m branch-2
git branch -m branch-1 branch-2
The name of a branch can be changed using this option. To change the name of the current branch only the new name is requried, if the target branch is not selecet the oldname and new name are required in that order.

git merge branch-2
Once additions have been made to a branch it can be merged back into the master. This is done by calling the command merge while the branch you wish to merge into is selected.

git branch branch-2 -d
git branch branch-2 -D
After the branch has been merged into the original it can be deleted. The option -d will delete the branch as long as it is called while on a different branch to the target. If the branch to be deleted is not fully merged the option -D may be used for force the deletion ('-d -f' should do the same as -D).

Config

The command 'git config' is used to change and specify configuration settings which manage certain features of git.

The config command will alter values in one of three files, depending on whether system, global or local was stated. These three refer to three settings files which each have a different priority level.

The local file belongs to a single repository and will only influence the settings while working in that repository. This file has the highest priority.
The global/user file defines the settings of the user that is currently in control. This will override system settings but is overridden by local settings.
The system file defines the settings of the current machine in use, which apply to all users. This will be overridden by all other settings files.

The following examples will use global but local or system may be used in it's place.

git config --list
This will print a list of the settings that are in effect

git config --global user.name "name"
git config --global user.email "e-mail"
These options define user details, which are applied to all commits made by the current user.

git config --global core.editor "editor name"
This option sets the text editor that should be used when entering commit messages.
Note: Take care with this command as using it incorrectly may result in issues with the program opening. It is easy to fix but better to get correct the first time. Click here for examples

git config --global commit.template full-file-path/message.txt
This allows you to set a template for any commit you make. The text provided in the file will be included in any commits you make (unless the message is written in the terminal using -m). You can include commented out text using the hash symbol as well.

git config --global core.pager ''
git config --global core.pager less
git config --global core.pager more
The pager is what is used to output information when commands such as diff or log are used. Using this command allows you to alert how the output is presented.
'' = print all information to the terminal
less = print a section of information (equal to the size of the terminal) and remove the information when the pager is closed (by pressing 'q' or 'ctrl+z')
more = print a section of information (equal to the size of the terminal) and leave the information in the terminal when the pager is closed (by pressing 'q' or 'ctrl+z')

Common Commands: commit

The commit command is used to recored changes to the repository.

git commit
Calling the base command 'git commit' will store all saved changes to the repository. The user is presented with a file so that a message describing the commit can be written.

git commit -m 'commit description'
git commit --message 'commit description'
This will commit all changes to the repository but rather than providing a file the commit description message it allows it to be entered into the terminal directly.

git commit -a
git commit --all
The all options allows you to commit modifications to files without staging them first (this is done for you). It will not commit untracked files however.

git commit -p
git commit --patch
Works in a similar way to git commit --all but allows you to select the specific changes in each file to commit.

git commit --amend
The option amend allows you to override the previous commit rather than creating a new one.

Common Commands: reset

This command is used to unstage changes in the repository and, if certain options are used, will also revert changes to files.

git reset
git reset file.txt
git reset file.txt file1.txt
The basic command will unstage all changes that are set to be included in the next commit. Specific files can be named to be reset.

git reset --soft HEAD^
git reset --soft HEAD~2
git reset --soft HEAD <first seven digits of SHA>
If the option soft is used with reset committed changes will be removed from the repository. If the symbol ^ is used changed on the previous commit will be undone. If the symbol ~X is used all changes from the previous X number of commits will be undone. This will not alter you files and will only remove commits. The first seven digits of a commits SHA may also be provided in order to target a specific commit.

git reset --hard HEAD^
git reset --hard HEAD~2
git reset --hard <first seven digits of SHA>
This is the equivalent of git reset --soft only any reverted changes will also effect your files.

Common Commands: add

This command is used to stage new or modified files. When commit is called the staged files current states will be updated in the repository. Add may be called numerous times before the staged changes are committed.

git add file.txt
git add file.txt file1.txt
git add folder/file.txt
Add must always be followed by additional information to instruct git on what action to take. The simplest instruction is the path to the file to be staged (the full file path should be provided). As shown it is possible to provide multiple files to be staged.

git add *
Add may be used with a asterisk, which will identify all files in the current directory and stage their changes. As deleted files are no longer in the directory the deletion will not be staged.
Note: The asterisk is not part of git but is a globbing operator which returns all file names within the working directory. This can be used with commands such as ls * and is supported by most shells

git add .
Using a dot character after git add works in a similar way to the asterisk. The difference is it will add all files to the staging area even if they are deleted or hidden files.
Note: In my experience the previous paragraph is true but information on the web seems to say that git add . will not stage deleted files.
git add . vs git add *
Difference between “git add -A” and “git add .”

git add -u
git add --update
Using -u or --update with git add will stage all changed made to files but will not stage new files.

git add -A
git add --all
git add --no-igonore-removal
The command git add -A will stage all files.

git add 'command' -v
git add 'command' --verbose
The verbose command is used to print details on the actions that are taken when other commands are called. For example if the command 'git add -A' is called all files will be staged but the terminal will not report this until git status is called. If 'git add -A -v' is called the terminal will print out a summary of the results.
git add -p
git add --patch
The command -p enables you to selectively stage changes with files rather than staging the entire file. This can be done on all files by calling the basic command (git add -p) or on specific files by providing a file name (git add file1.txt -p).

Common Commands: status

git status

This command displays files that have been altered or removed since the last commit. The first set of files are those that are staged and will be to be added or updated on the next commit.

When called the terminal will display the name of the current working branch, the number of commits the local repository is ahead of the remote repository (if a remote repository is connected), the staged changes that will be added on the next commit, unstaged changes and finally any untracked files.

Options that may be used with the command status include...

git status -s
git status --short
This produces a quick summary of the state of all the files in the directory. The state of each file is identified using single character: M = modified, A = added, D = deleted, R = renamed, C = copied, U = updated but unmerged, ? = untracked.

git status -s -b
git status -s --branch
This provides a summary of the current branch along with the summarised file states.

git status -v
git status --verbose
Along with the normal output of the command it produces the results of the command git diff (presents details of the changes made within each file)

Additional options: git-status documentation.

Common Commands: init

git init

This command is used to create a new repository in the current working directory. A subdirectory named .git will be created in the directory, which is the basics required for a repository. No files contained in the directory will be tracked by git until they are added and committed.

This command may also be used to reinitialise a current repository.

What Is Git

Git is an open source version control system developed by the creator or Linux (Linus Torvalds). Version control is a method of recording changes to a set a files over a given period of time and allows the user to roll back the contents to a previous state if required. The details on all changes made are stored in a data structure known as a repository. A repository on the users machine is known as a local repository. If the files are hosted by a server, which allows for collaboration on a project, it is known as a remote repository.

Git is a distributed version control system (as opposed to a centralised version system) which means when clients checkout the files from a remote repository all of the changes are also provided, meaning the new user has access to the full file history.

Git works by recording snapshots of files. On each commit the state of each file that has changed is recorded. If a file has not changed a link to a previous version is provided to save space.
